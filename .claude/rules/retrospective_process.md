# 研究复盘与知识库更新规则

## 核心原则

每次完成一轮研究、实验、或重要发现后，**必须**做一次自我审查复盘，并把关键学习更新到知识库。不要只往前冲，不要累积隐性经验。

## 什么算一次"研究或发现"

触发复盘的情形：
- 查阅/下载文献后得到新信息（如 benchmark 调研、相关工作对比）
- 训练/实验完成后拿到结果（不论成败）
- 和用户讨论后论文方向发生变化
- 调试出一个 bug 或技术难点（dtype/OOM/兼容性）
- 发现被别人已经做过（concurrent work）

## 复盘应该回答的 5 个问题

1. **出发前的假设是什么？**（在做之前我以为会怎样）
2. **实际发现了什么？**（数字、现象、反例）
3. **哪里被打脸了？**（假设错在哪）
4. **这改变了什么决策？**（论文角度/方法/下一步）
5. **有没有被重复浪费的时间？**（下次怎么避免）

## 更新知识库的规则

### 什么该记录
- ✅ **数字**（benchmark 分数、val loss、延迟）
- ✅ **反直觉的发现**（Base > LoRA、数据多了更差）
- ✅ **技术坑**（MPS dtype、Ollama thinking mode）
- ✅ **并行工作**（arXiv 编号 + 核心贡献 + 差别）
- ✅ **论文角度迭代**（v1→v2→v3 的演化和原因）

### 什么不该记录
- ❌ 实现细节（代码里有）
- ❌ 一次性的调试日志
- ❌ 临时状态（训练进度）

### 文件组织


## 复盘模板

每次更新知识库用这个格式：

```markdown
## [日期] [主题]

### 背景
（之前在做什么，为什么开始这次研究）

### 发现
（关键数字、链接、反直觉事实）

### 假设 vs 现实
（原本以为 X，实际发现 Y）

### 影响
（改变了哪些决策）

### 下次注意
（可操作的经验，下次少走弯路）
```

## 反模式（要避免的）

1. **只记成功不记失败**：LoRA 过拟合、3B OOM、thinking mode 污染都是重要教训
2. **不记并行工作**：Fixed-Persona SLMs (arXiv 2511.10277) 和我们几乎同方向，不记录会重复踩坑
3. **把复盘当日志**：复盘要提炼 *lessons*，不是时间线
4. **复盘后不更新**：审查完不回写知识库 = 白审查

## 执行时机

- **不要**：每写一行代码就复盘
- **要**：一个"研究单元"完成就复盘。比如：
  - 一轮 benchmark 跑完
  - 一个新论文调研完
  - 一次训练完成（不论成败）
  - 用户指出方向错误后

## 复盘的长度

- 技术坑：100-200 字足够（问题 + 解法 + 避坑要点）
- 论文方向变更：300-500 字（前后对比 + 依据）
- Benchmark / 并行工作发现：500-1000 字（完整元数据）
